## 2. 연속가동 시상 상세 조건

### 1~2월 연속가동 (1주차 4번)

- **1월 구간**: 1월 19일 ~ 1월 31일 실적이 **10만 / 20만 / 30만 / 50만** (AND)
- **2월 구간**: 2월 1일 ~ 2월 18일 실적 **10만원 이상**
- **시상금**: 20만, 60만, 80만, 180만원

### 1~2월 추가 연속가동 (1주차 5번)

- **1월 구간**: 1월 26일 ~ 1월 31일 실적이 **10만 / 20만 / 30만 / 50만** (AND)
- **2월 구간**: 2월 1일 ~ 2월 18일 실적 **10만원 이상**
- **시상금**: 20만, 45만, 80만, 180만원

### 2월~3월 연속가동 (3주차 3번)

- **2월 구간**: 2월 19일 ~ 2월 28일 실적이 **10만 / 20만 / 30만 / 50만** (AND)
- **3월 구간**: 3월 1일 ~ 3월 15일 실적 **10만원 이상**
- **시상금**: 20만, 60만, 80만, 180만원

### 2월~3월 추가 연속가동 (3주차 4번)

- **2월 구간**: 2월 19일 ~ 2월 28일 실적이 **10만 / 20만 / 30만 / 50만** (AND)
- **3월 구간**: 3월 1일 ~ 3월 8일 실적 **10만원 이상**
- **시상금**: 20만, 60만, 80만, 180만원

### 2.2 연속가동 시상금 구간 매핑 (해당 금액 구조)

연속가동 시상은 **선행 구간 실적(10/20/30/50만원)**을 달성한 뒤, **다음 기간에 10만원 이상** 달성 시 해당하는 시상금을 받는 구조입니다.

| 선행 구간 실적 (만원) | 다음 기간 10만원 이상 달성 시 시상금 |
|----------------------|--------------------------------------|
| 10만                 | 20만원                               |
| 20만                 | 60만원                               |
| 30만                 | 80만원                               |
| 50만                 | 180만원                              |

- **해당 금액 구조**: 선행 구간 **10만** 달성 후 다음 기간 10만 이상 → **20만원**, **20만** 달성 후 → **60만원**, **30만** 달성 후 → **80만원**, **50만** 달성 후 → **180만원** 시상.
- **1~2월 연속가동, 2~3월 연속가동, 2~3월 추가 연속가동**: 위 표 적용 (10→20, 20→60, 30→80, 50→180).
- **1~2월 추가 연속가동**만 20만 구간 시상금이 **45만원**으로 다름:

| 선행 구간 실적 (만원) | 다음 기간 10만원 이상 달성 시 시상금 (1~2월 추가만) |
|----------------------|---------------------------------------------------|
| 10만                 | 20만원                                            |
| 20만                 | **45만원**                                        |
| 30만                 | 80만원                                            |
| 50만                 | 180만원                                           |

### 2.3 연속가동 시상 상세 로직 (전문)

아래는 각 연속가동 시상의 **달성 조건**과 **시상금**을 기간·실적 기준으로 풀어서 정리한 내용입니다. 시상금은 §2.2의 구간 매핑(10→20, 20→60, 30→80, 50→180 등)을 따릅니다.

#### 12~1월 연속가동

- **달성 조건**: **1월 18일까지의 실적이 10만원 이상**이면 달성.
- **시상금**: 달성 시 해당 시상금 표시. 구간에 따라 **20만원 / 60만원 / 80만원 / 180만원** 중 해당 금액 지급.
- **대시보드 표시**: 1월 18일까지 10만원 이상이면 “달성”으로 표시하고, 해당 시상금(20/60/80/180만원)을 노출.

#### 1~2월 연속가동

- **1단계(선행 구간)**: **1월 19일 ~ 1월 31일** 기간 실적이 **10만원 / 20만원 / 30만원 / 50만원** 구간 중 하나를 달성해야 함.
- **2단계(완성 조건)**: **2월 1일 ~ 2월 18일** 기간 실적이 **10만원 이상**이면 시상금 확정.
- **시상금**: 위 두 조건을 모두 만족할 때, 1월 구간 달성 수준에 따라 **20만원 / 60만원 / 80만원 / 180만원** 중 해당 금액 지급.
- **대시보드 표시**: 1월 구간(1/19~1/31) 실적 기준 10/20/30/50만 게이지 표시, “2월 18일까지 10만원 달성 시 예상 시상금” 문구와 함께 위 시상금 표시.

#### 1~2월 추가 연속가동

- **1단계(선행 구간)**: **1월 26일 ~ 1월 31일** 기간 실적이 **10만원 / 20만원 / 30만원 / 50만원** 구간 중 하나를 달성해야 함.
- **2단계(완성 조건)**: **2월 1일 ~ 2월 18일** 기간 실적이 **10만원 이상**이면 시상금 확정.
- **시상금**: 위 두 조건을 모두 만족할 때, 1월 구간(1/26~1/31) 달성 수준에 따라 **20만원 / 45만원 / 80만원 / 180만원** 중 해당 금액 지급.
- **대시보드 표시**: 1월 구간(1/26~1/31) 실적 기준 10/20/30/50만 게이지 표시, “2월 18일까지 10만원 달성 시 예상 시상금” 문구와 함께 위 시상금 표시.

#### 2~3월 연속가동

- **1단계(선행 구간)**: **2월 19일 ~ 2월 28일** 기간 실적이 **10만원 / 20만원 / 30만원 / 50만원** 구간 중 하나를 달성해야 함.
- **2단계(완성 조건)**: **3월 1일 ~ 3월 15일** 기간 실적이 **10만원 이상**이면 시상금 확정.
- **시상금**: 위 두 조건을 모두 만족할 때, 2월 구간(2/19~2/28) 달성 수준에 따라 **20만원 / 60만원 / 80만원 / 180만원** 중 해당 금액 지급.

## 3. 원본 엑셀(PRIZE_SUM) 컬럼 매핑

파트너 시상 데이터는 **일별 시상 집계 엑셀**에서 읽어옵니다.  
파일 경로 예: `data/daily/{MMDD}PRIZE_SUM_OUT_202602.xlsx` (예: `0227PRIZE_SUM_OUT_202602.xlsx` → 2월 27일 기준 2026년 2월 회차).

### 3.1 실적 매핑 원칙

- **월간 실적·전월 실적·전전월 실적·주차별 실적**은 **기존 mclist(대시보드) 데이터**를 그대로 사용합니다.  
  - Appwrite `agents` 문서의 `performance`(월별), `weekly`(주차별) 필드가 해당합니다.  
  - PRIZE_SUM 엑셀 업로드는 **시상금·연속가동 구간 실적**만 `partner` 필드로 보강하며, 위 기본 실적을 대체하지 않습니다.

### 3.2 파일 규칙

| 항목 | 설명 |
|------|------|
| **파일명 패턴** | `{MMDD}PRIZE_SUM_OUT_202602.xlsx` |
| **의미** | 날짜(MMDD) + PRIZE_SUM + 회차(연월) |
| **키 컬럼** | **K열: 설계사코드** → 이 코드로 매니저·실적·설계사 정보 매칭 |

### 3.3 2월 데이터 컬럼 매핑 (0227PRIZE_SUM_OUT_202602.xlsx 기준)

아래는 **2월** PRIZE_SUM 엑셀의 열(알파벳)과 값(내용) 매핑입니다. 업로드 스크립트는 이 열 인덱스로 읽어 `agents.partner` JSON에 저장합니다.

| 엑셀 열 | 값(내용) | agent.partner 필드 |
|--------|----------|---------------------|
| **K** | 설계사코드 | 매칭 키 (code) |
| **R** | 실적 1주 | (참고용, 주차 실적은 mclist/weekly 사용) |
| **S** | 실적 2주 | (참고용) |
| **T** | 실적 3주 | (참고용) |
| **U** | 실적 4주 | (참고용) |
| **AB** | 1주차 인보험 시상금 | `productWeek1InsPrize` |
| **AC** | 1주차 상품 실적 | `productWeek1` |
| **AD** | 1주차 상품 시상금 | `productWeek1Prize` |
| **AG** | 2주차 인보험 시상금 | `productWeek2InsPrize` |
| **AH** | 2주차 상품 실적 | `productWeek2` |
| **AI** | 2주차 상품 시상금 | `productWeek2Prize` |
| **AL** | 3주차 인보험 시상금 | `week3Prize` |
| **AM** | 3–4주차 합산 실적 | `week34Sum` |
| **AN** | 3–4주차 인보험 시상금 | `week34Prize` |
| **BL** | 1–2월 연속가동 실적 (1월) | `continuous12Jan` |
| **BM** | 1–2월 연속가동 실적 (2월) | `continuous12Feb` |
| **BN** | 1–2월 연속가동 시상금 | `continuous12Prize` |
| **BR** | 1–2월 추가 연속가동 실적 (1월) | `continuous12ExtraJan` |
| **BS** | 1–2월 추가 연속가동 실적 (2월) | `continuous12ExtraFeb` |
| **BT** | 1–2월 추가 연속가동 시상금 | `continuous12ExtraPrize` |
| **BX** | 2–3월 연속가동 실적 (2월) | `continuous23Feb` |

### 3.4 업로드 시 사용 흐름

1. `{MMDD}PRIZE_SUM_OUT_202602.xlsx` 읽기 → **K열**로 설계사(code) 식별.
2. **AB, AC, AD** → 1주차 인보험 시상금, 상품 실적, 상품 시상금 → `productWeek1InsPrize`, `productWeek1`, `productWeek1Prize`.
3. **AG, AH, AI** → 2주차 인보험 시상금, 상품 실적, 상품 시상금 → `productWeek2InsPrize`, `productWeek2`, `productWeek2Prize`.
4. **AL, AM, AN** → 3주차 인보험 시상금, 3–4주차 합산 실적, 3–4주차 인보험 시상금 → `week3Prize`, `week34Sum`, `week34Prize`.
5. **BL, BM, BN** → 1–2월 연속가동 구간 실적·시상금 → `continuous12Jan`, `continuous12Feb`, `continuous12Prize`.
6. **BR, BS, BT** → 1–2월 추가 연속가동 구간 실적·시상금 → `continuous12ExtraJan`, `continuous12ExtraFeb`, `continuous12ExtraPrize`.
7. **BX, CB** → 2–3월 연속가동 2월 실적, 2–3월 추가 연속가동 2월 실적 → `continuous23Feb`, `continuous23ExtraFeb`.

필드명: `productWeek1`, `productWeek1InsPrize`, `productWeek1Prize`, `productWeek2`, `productWeek2InsPrize`, `productWeek2Prize`, `week3Prize`, `week34Sum`, `week34Prize`, `continuous12Jan`, `continuous12Feb`, `continuous12Prize`, `continuous12ExtraJan`, `continuous12ExtraFeb`, `continuous12ExtraPrize`, `continuous23Feb`, `continuous23ExtraFeb` (JSON 객체를 `partner` 문자열 속성으로 저장)

### 3.5 Appwrite 컬렉션 설정

- **agents** 컬렉션에 속성 추가: **partner** (타입: string, 크기 65535 등 충분한 크기).  
  콘솔 → Databases → 해당 DB → agents → Attributes → Create attribute → String → `partner`

### 3.6 업로드 스크립트 실행

```bash
# 프로젝트 루트(afg-dashboard)에서
node scripts/upload-partner-prize.js
# 또는 파일 지정
node scripts/upload-partner-prize.js "C:\...\data\daily\0227PRIZE_SUM_OUT_202602.xlsx"
```

파일 경로를 생략하면 `data/daily` 폴더에서 최신 `*PRIZE_SUM*.xlsx` 파일을 자동 선택합니다.

## 4. 추가로 필요한 데이터 파일

시상 계산·대시보드 반영을 위해 아래 두 종류의 원장/파일이 필요합니다.

### 4.1 연속가동 관련 파일

- **역할**: 구간별(일 단위 또는 주 단위) 실적로 연속가동 조건 충족 여부 판단
- **포함 필드 예시**:
  - 설계사 식별자(code 등)
  - 구간 시작일·종료일 (또는 구간 코드)
  - 해당 구간 실적 금액
- **용도**: “1/19~1/31”, “2/1~2/18”, “2/19~2/28”, “3/1~3/15” 등 구간별 합산 후 시상 매칭

### 4.2 상품별 실적 정리 파일

- **역할**: 인보험 / 통합·간편·어린이 / 메리츠클럽+ 등 **상품군별** 실적 집계
- **포함 필드 예시**:
  - 설계사 식별자
  - 상품 구분 (인보험, 통합, 간편, 어린이, MC+ 등)
  - 주차 또는 회차
  - 실적 금액 (또는 건수·금액)
- **용도**: 1주차/2주차/3주차별, 상품별 시상 테이블(10→20, 20→40 등)과 매칭

## 5. 원장 정리 및 매일 업로드 계획

### 5.1 원장 구성 원칙

1. **단일 원장(마스터) 유지**
   - 연속가동: `연속가동_원장.csv` (또는 시트 1개)  
     - 컬럼: code, 구간명, 시작일, 종료일, 실적금액, 기준일(업로드일) 등
   - 상품별 실적: `상품별실적_원장.csv` (또는 시트 1개)  
     - 컬럼: code, 상품구분, 회차/주차, 실적금액, 기준일 등
2. **일자/회차 규칙 고정**
   - 연속가동 구간(1/19~1/31, 2/1~2/18 등)은 규칙 문서와 동일한 날짜로 정의
   - 상품별 실적은 “13회차”, “1주차/2주차/3주차” 등 기존 회차 체계와 1:1 매핑
3. **매일 갱신 방식**
   - 전일(또는 당일) 마감 데이터만 추가·갱신하는 **증분 파일**을 만들고,
   - 이를 원장에 **append** 또는 **key(설계사+구간/회차) 기준 upsert**하여 “매일 업데이트” 형태로 운영

### 5.2 업로드 주기 및 흐름

| 단계 | 주기 | 내용 |
|------|------|------|
| 1 | 매일 (정해진 시간 1회) | 연속가동 구간별 일 실적 또는 일별 집계 반영 |
| 2 | 매일 (동일 또는 별도 배치) | 상품별 실적(인보험, 통합·간편·어린이, MC+ 등) 반영 |
| 3 | 업로드 후 | Appwrite(또는 현재 백엔드)에 반영 → 대시보드에서 “파트너” 지사만 위 규칙으로 시상 계산 |

### 5.3 기술적 반영 방안 (선택)

- **옵션 A**: 기존처럼 **스크립트로 CSV/Excel 읽어서 Appwrite에 업로드**  
  - 예: `scripts/appwrite-upload-daily.js` 확장 또는 `scripts/upload-partner-continuous.js`, `upload-partner-product.js` 추가
- **옵션 B**: 연속가동·상품별 실적을 **Appwrite 컬렉션(문서)** 로 보관  
  - 설계사 문서에 “연속가동 구간별 실적”, “상품별 실적” 필드(객체/배열) 추가 후 일일 배치가 해당 필드만 갱신
- **옵션 C**: 별도 **연속가동 전용 컬렉션**, **상품별 실적 전용 컬렉션**을 두고, 대시보드/시상 로직에서 조인해서 사용

## 6. 발생 가능한 문제점 및 대응

### 6.1 성능·렉 (Performance)

| 문제 | 원인 | 대응 |
|------|------|------|
| 업로드/동기화 느림 | 한 번에 많은 문서 쓰기, 대용량 CSV 파싱 | 배치 단위 분할(예: 100~200건씩), 비동기 처리 또는 큐 도입 |
| 대시보드 느림 | 파트너 시상 계산이 페이지 로드 시 실시간 집계 | 연속가동·상품별 시상은 **사전 집계 필드**로 저장하고, 일일 배치에서만 계산 후 문서에 저장 |
| 조회 시 지연 | 연속가동/상품별 데이터를 매번 여러 컬렉션에서 조인 | 필요한 구간·회차만 넣은 “파트너 시상 요약” 문서 또는 뷰를 두고, 대시보드는 해당 문서만 조회 |

### 6.2 한도 (Rate limit / Quota)

| 문제 | 원인 | 대응 |
|------|------|------|
| Appwrite 쓰기 한도 초과 | 일일 업로드 건수가 많을 때 문서 생성/업데이트 제한 | 요청 간 딜레이(delay), 배치 크기 축소, 가능하면 “업데이트 위주”로 설계해 새 문서 수 최소화 |
| API 호출 제한 | 동시에 많은 읽기/쓰기 | 업로드 스크립트에서 동시 요청 수 제한(예: 5~10개), 재시도 + 지수 백오프 |
| 스토리지/문서 크기 | 문서당 필드가 과도하게 커짐 | 연속가동·상품별 실적은 “요약 구조”(구간코드 + 금액)만 저장하고, 상세는 별도 컬렉션 또는 파일로 유지 |

### 6.3 데이터·운영

| 문제 | 원인 | 대응 |
|------|------|------|
| 구간 경계 오류 | 날짜가 “1/19~1/31” 등과 1일 차이로 잘못 집계 | 원장 스키마에 “구간코드”(예: `2026-01-19_2026-01-31`)를 명시하고, 규칙 문서와 매핑 테이블로 검증 |
| 상품 구분 오류 | 인보험/통합·간편·어린이 구분이 원본과 다름 | 상품별 실적 원장에 “상품코드” 또는 “상품명” 정의서를 두고, 업로드 전 검증 로직 추가 |
| 중복 반영 | 같은 날 두 번 업로드 시 이중 집계 | “기준일 + 설계사 + 구간/회차”를 유니크 키로 하고, upsert(있으면 업데이트, 없으면 생성)로 처리 |
| 지사 구분 누락 | “파트너”가 아닌 지사에 파트너 시상 적용 | 지사명(또는 branch)에 “파트너” 포함 여부를 필드로 두고, 시상 계산·표시 분기 시 반드시 사용 |

### 6.4 스키마·버전

| 문제 | 원인 | 대응 |
|------|------|------|
| 구간/회차 추가·변경 | 13회차 이후, 또는 새 연속가동 구간 추가 | 규칙 문서(본 md)와 원장 컬럼/구간코드를 버전으로 관리하고, 스크립트는 “구간 정의 테이블”을 참조하도록 구성 |
| 과거 데이터 정합성 | 이전 회차 데이터 수정 시 시상 재계산 필요 | “재계산 배치”를 별도로 두고, 필요 시 과거 구간만 선택 실행 |
